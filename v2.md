# General Format

The OOD Data format consists of __specifiers__ and __values__.  
While the actual data is presented by __Values__ the __Specifiers__ provide the neccesary information to parse the Data.  
__Data__ and __Values__ will be used as synonyms.
___
The document structure is json-like with nested objects and values. No default type will be used. If the parsed data is longer than the actual data that is specified, all excess data will be ignored.

# Value Format and Syntax

The Values have individual formats depending on their type specifier.

# Specifiers & Headers

Note: It is not safe to manually access the specifiers. Although the TypeSpecifier and type are always correct, it is not safe that a complexSpecifier is up to date. It is only updated upon saving.  
Every Element consists of a __Header__, followed by __Data__.  
A header consists of a list of specifiers.  
Not all specifiers are part of the header, they can also be part of the __Data__, in case of a nested Object.  
If multiple specifiers are needed they will be saved by their priority:

1. Type
2. Size (Array)
3. Fixtype (for Fixed types)
4. Object

The size specifier must come directly after the type specifier so the laod and save Functions of it can just peek the previous byte and modify it with the sizeSpecSpec

## Type Specifier

Every __Header__ starts with a __type specifier__.  
It specifies the type of the element. Depending on its type, additional specifiers or Values will follow.  
If the type specifier is wrong (especially if the sizes of the declared and actual type does not match) bugs occur and following specifiers can be interpreted as __Values__.

See below for a List of Type Specifiers

## Object Specifier

An Object Specifier is needed by every [Object Type](#Object-Types).
An Object Specifier ist a specifies what properties and fields are contained in an object.  
It is a part of the element header.

# Actual Definitions

## linking and typedefs

For sharing types across files, the runtime uses scopes.  
All types defined with deftype are only available in the local scope (the current file).

For sharing types across files, the concept of pools is intruduced.  
A pool is a collection of files, that is not defined by the files included in it, but instead by the program that loads the file.  
It can create a pool and add files to it. If no pool is specified, the global pool will be used.  
An IMPORT declaration can possibly import all types available in the current pool.  
Multiple pools can be used at once, although an error occurs if the same type id is defined twice.
An error also occurs if a requested type ID is not present at all.
A file can Export to pools that are different from the pools it imports from to create more complex dependencies.  
To which pools a file exports or from which pools it imports is specified by the program that parses the file. A possible solution for using a purely file based approach is to write the pool informations into another file, that uses no pool or the global pool.
Every File is in the global pool by default and will be removed from it once it is either explicitely removed from it, added to another pool or makes an error by exporting a already present type ID.

The Types A and B are used for declaring and using custom types, as well as importing and exporting them.  
The Extra bit specifies wether the declaration references to a type from this file or from another file.

### DEFTYPE (DEFINE TYPE)

DEFTYPE is a virtual type, which means that it isn't really a usable type, but instead holds data on how the other type should be interpreted.  
DEFTYPE is also a size specified type.  

Depending on the type from the small size specifier, the TypeID is 0, 1, 2 or 4 bytes long.
If it is 0 bytes long, the unused codes from the normal type byte will be allocated for the use as custom type.  
Otherwise the USETYPE virtual needs to be used to use a previously defined type.  
Internally this is saved in a uint64. If the most significant byte is all 1s (0xFF) the last significant byte will be treated as a potential replacement for a built-in type, otherwise the first 32 bits are ignored and the 4 least significant bytes serve as 32-bit type id.

Format:
```
DEFTYPE
ID
TYPE
```

### USETYPE

Uses a previous defined type that has a size of 1, 2 or 4, as the DEFTYPE can also use existing non-allocated type codes.  
Note that USETYPE needs to specify the same size specifier which the type was Defiend with. Otherwise a Status::InvalidLocalTypeID will be throw, as it will not be found.

Format:
```
USETYPE
ID
// body...
```

### EXPORT

Exports a previously (with DEFTYPE) defined type into all current pools.

Format:
```
EXPORT
ID
```

### EXPORTAS

Exports a previously defined type (with typeID 'LOCALID') into all curretn pools, but via the typeID 'GLOBALID'.

Format:
```
EXPORTAS
LOCALID
GLOBALID
```

### IMPORT

works just like USETYPE but uses an imported type. Imports the Type with TypeID 'ID' that was exported by another file in the currently pool (that file needs to be parsed before the current one to simplify implemetation and avoid circular dependencies)

Format:
```
IMPORT
ID
```

### IMPORTAS

works like IMPORT, but the imported type (with TypeID 'GLOBALID') becomes available via 'LOCALID'.  
Both IDs have the same length, because the sizespecifier only has 2 bits available

```
IMPORT
GLOBALID
LOCALID
```

## Mixed Object Specifier

A mixed object Specifier will look as following:  
```
KEY
TYPE
```
where `KEY` is the key represented as CSTR (with NULL terminator !)   
and `TYPE` is the Type Header of the type of the corresponding value   
If the `KEY` is empty (just a null terminator) the object specifier ends.

## Fixed Object

A fixed type object specifier consist of a type header `TYPE` and a list of keys, represented as CSTRs.  
It ends with an empty key, like the `Mixed Object Specifier`   
```
TYPE
KEY1
KEY2
...
NULL

VALUE
VALUE
...
```

## Size Specifier

A Size Specifier is a 8, 16, 32 or 64 bit unsigned integer.  
It is used to specify a size in the corresponding element.  
It can for example specify an array size.

# Type Categories

## Primitive Types
- Signed and Unsigned Integers with 8, 16, 32 and 64 bit precision
- floating point and double precision floating point numbers
- standart strings, wide character strings

## Size specified Types (SSTs)

Size specified Types need some form of static size specifier.  
The 2 most significant bits of every Size Specified Type determine the size of the size specifier.  

|end bits|Size Spec. Length (in bits)|
|---|---|
|00|8|
|01|16|
|10|32|
|11|64|

This Length can be calculated with the function  
`size(bits) = 8 * 2**bits`  

The fixed array is the only type that uses a size specifier currently

***Implcit Type Specifiers***  
The OOD Implementation can implicitely select the smallest fitting size specifier, if, for example, an element is specified as `array` without size specifier.  
***The Following Types SSTs support implicit size:***  
- Arrays

### Array Types

Array Types include the Mixed and Fixed List / Array.
Only the fixed array is a size specified type. As in the mixed array every Element has its own type, it is more storage efficiet to just termiante the Array with a NULL type instead of storing a size specifier in addition to the actual data

Mixed Array structure:
```

TYPE
TYPE
...
NULL

VALUE
VALUE
VALUE
...
```

Fixed Array structure:
```

VALUE
VALUE

```

Costruction of Array Types:

```cpp

```


## Object Types

Object types include the Mixed and Fixed object. They store data in a key-value container.
As both objects can use an empty key to indicate the end of object, they both do not need a size specifier.

```

TYPE
KEY
TYPE
KEY

NULL
...

VALUE

```

## Automatic merging of fixed types

If a fixed type is constructed with an initializer list that holds multiple types, for example like `ODF::Array::FixedArray list = { 5ui16, 6ui32 }`,  
the implementation automatically converts those to the smallest possible type that still fits all elements without data-loss.  
Wether a conversion to a smaller or bigger type is happenening on float and string types is determined by the config flags `PrevetAutomaticMergeLossString` and `PrevetAutomaticMergeLossFloat`. Both of them Prevent the loss of data through automatic merge of types.

## Automatic conversions

If operator[] is used on a primitive type, it is covnerted into the corresponding mixed complex type

### ConversionLossLevel

Specifies how much the size of a data type can shrink during a conversion to fit i a fixed element   
If an invalid conversion attempt occurs an InvalidConversionLevel exception is thrown

|value|meaning|
|---|---|
|0|Forbidden. No conversion allowed|
|1|Lossless. A conversion to a smaller type is valid if the conversion does not result in the change or loss of data, for example converting 5ui16 to 5ui8. Also allows conversions to larger types.|
|2|Unrestricted. Any conversion can happen event if it results in the change or loss of data.|

Allowed coversions:

|Conversions|Forbidden|Lossless|Unrestricted|
|:---|:---:|:---:|:---:|
|float -> double|❌|✅|✅|
|double -> float|❌|❌|✅|
|int -> smaller int|❌|✅|✅|
|int <-> float / double|❌|❌|❌|

If a complexType function is called on a wrong type, it will be implicitely converted to the correct one.  
Those functions currently include:
operator[](const std::string_view&) -> converted to MXOBJ
push_back -> converted to MXLIST
insert -> converted to MXLIST or MXOBJ

any other function, which cannot add anything to the complex type will just throw bad_variant_access, as it couldn't do anything to the just create complex object, because it would be empty

T at(...) also throws bad_variant_access
 

# Just some Implementation Details

The value is stored as an instance of a variant that holds all target types. `std::variant<char, uchar, .., std::wstring, ...>`  

## Yapping over saveToMemory with Pools

Using pools in saveToMemory does slightly differ from loadFromMemory. loadFromMemory just takes the file, resolves all virtual types and loads all non-virtual types.  
This doesn't work for saveToMemory as the implemetation must know somehow, which virtual types should be used.

Solution:

1. An ODF Object can be specified as TypeImmutable on loading. If that's the case, the implementation reproduces an exact copy of the Document, just with other element bodies on saving.  
This is done by making all Types immutable directly during the initial parsing phase.

2. Do not use virtual types. Doesn't uses any virtual types.

3. Create virtual types automatically. If this is the case, the implemetation searches through all elements in the document and checks in which cases the overall document size would decrease, if a type is made virtual.
All those virtual type will then be positioned at the start of the document.

4. Use an existing Pool Collection. In this case, an existing PoolCollection is provided. The implementation automatically imports every needed type that is already existing.

5. Explicit virtual types. In this case, a struct that contains information about the virtual types is passed to saveToMemory. The struct contains all information, including exported or imported types and which types should use USETYPE or be defined.

Implemetation:

ODF Fíles are designed manually or with the ODF-Designer.  
On loading, the vtypeinfo property of the root odf object is filled and holds all neccessary information for all virtual types.  
It can be modified at runtime.  
The provided data is used for saving.  
Every Datatype that is already defined somehow will be automatically used.


## Complex Type handling

The complex types are `MixedObject`, `FixedObject`, `MixedArray` and `FixedArray`   
They all derive their member variables regarding anything fixed/mixed handling from the `AbstractType` base class.   
The Object Types are wrapped inside of an `Object` class and the Array types in an `Array` class.   
The `Object` and `Array` types are __forwarded Types__ as they just wrap around a `std::variant<FixedType,MixedType>` and forward all regular calls to their wrapped objects.   
No function of the `AbstractType` class is actually implemented, they all get implemented by their Inheritors (`Object` and `Array`).

### Member Functions of `AbstractType` Objects

#### virtual void makeMixed() = 0

Converts the content to a mixed Type. ignored if content already is mixed

#### virtual bool canBeFixed() = 0

Returns true if the content could be converted to a fixed type (That means every child object has the same value).
This functions checks recursively if child objects are ComplexTypes

#### virtual bool tryFixing() = 0

Converts the Mixed object into a fixed one. If not all child objects have the same type it returns false.
True is returned on success.
If the Object only contains integers, but made of different types, it picks the smallest type still fits all childs.
If the Object contains floating points, it picks the floating point type with the highest precision

#### virtual Type getFixedDatatype() = 0

***Only for Fixed Objects!***

Returns the datatype

#### virtual Type getTargetDatatype() = 0

***Only for Mixed Objects***

Returns the datatype that would be used if the Object would be converted to a fixed object.   
Returns NULL if fixing is not possible.
